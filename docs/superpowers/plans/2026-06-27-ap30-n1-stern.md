# AP-30 N-1-Stern Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ein Start zieht in einem read-only SELECT Attribute aus mehreren Lookup-Tabellen (N-1-Stern), indem die Tabellen ausgewählter Spalten/ORDER-BYs/Filter automatisch in den Join-Baum gewebt werden; absteigende (1-N) Äste lösen eine nicht-blockierende Warnung aus.

**Architecture:** Die Weaving-Infrastruktur existiert bereits (`find_paths` webt Pflicht-Tabellen als Baumäste, `generate_sql` rendert Multi-Tabellen-Selects). Diese Scheibe (a) verallgemeinert den Pflicht-Tabellen-Parameter und füttert ihn aus der Vereinigung Filter ∪ Select ∪ ORDER BY, (b) entfernt das stille Pro-Pfad-Verwerfen, (c) macht die FK-Richtung als `JoinStep.to_many` sichtbar und (d) zeigt daraus abgeleitete Warnungen pro Pfad an.

**Tech Stack:** Python 3.14 (venv via uv), Flask, SQLAlchemy, NetworkX; vanilla JS Frontend; pytest.

## Global Constraints

- **Layering:** `core/` darf **niemals** Flask importieren. Web (`web/`) ruft `core/`, nie umgekehrt.
- **Read-only:** Es wird nur Schema-Metadaten gelesen und SQL-Text erzeugt; keine Mutation.
- **No CDN:** Keine externen `<script src="https://…">`/`<link href="https://…">`; alle Assets lokal unter `web/assets/`.
- **Determinismus:** Gleiche Eingabe → identische SQL. Jede neue Tabellen-Iteration muss deterministisch geordnet sein (kein `set`-Iterieren über Pflicht-Tabellen).
- **Version:** `config.APP_VERSION` **nie** von Hand ändern — nur via `./venv/bin/python sync_version.py`. Diese AP ist ein Feature → `--minor`.
- **Commits:** je AP committen, **ohne** KI-Signatur.
- **Tests laufen via** `./venv/bin/python -m pytest`. Baseline aktuell: 138 passed, 1 skipped.

```python
@dataclass(frozen=True)
class JoinStep:
    left_table: str
    right_table: str
    # One (left_col, right_col) pair per key column; >1 pair = composite FK,
    # all combined with AND in the emitted JOIN ... ON clause.
    column_pairs: tuple[tuple[str, str], ...]
    # True when this step descends a foreign key (left=parent -> right=child,
    # i.e. right holds the FK): a one-to-many edge that can multiply rows.
    to_many: bool = False
```

- [ ] **Step 4: `_join_step` auf Options-Auswahl + Orientierung umstellen**

```python
    options = graph[a][b]["joins"]
    # Deterministic choice: the option whose a-oriented pairs sort smallest.
    chosen = min(options, key=lambda o: _oriented_pairs(o, a))
    pairs = _oriented_pairs(chosen, a)
    # The chosen FK is held by chosen.table_a (the child/many side). Stepping
    # a -> b descends (one-to-many) exactly when b is that FK holder.
    to_many = chosen.table_a == b
    return JoinStep(a, b, pairs, to_many)
```
